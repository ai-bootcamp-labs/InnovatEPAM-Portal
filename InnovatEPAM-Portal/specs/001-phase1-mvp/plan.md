# Implementation Plan: Phase 1 MVP — InnovatEPAM Portal

**Branch**: `001-phase1-mvp` | **Date**: 2026-05-12 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/001-phase1-mvp/spec.md`

## Summary

Deliver the Phase 1 MVP of the InnovatEPAM Portal: a web application that lets
EPAM staff (Submitters) register, log in, and submit innovation ideas with a
single optional file attachment, and lets Admins triage those ideas through a
four-state evaluation workflow (Submitted → Under Review → Accepted | Rejected)
with mandatory decision comments and a full audit trail.

Technical approach is fixed by the project constitution: a single .NET 8 ASP.NET
Core Web API (controllers + EF Core 8 + PostgreSQL 15, ASP.NET Core Identity for
auth) and a single React 18 + TypeScript SPA (Vite, Tailwind, shadcn/ui,
TanStack Query, React Router, react-hook-form + zod, **date-fns** for date
formatting). Attachments are stored on the local filesystem behind an API
endpoint in Phase 1; the storage abstraction allows swapping to Azure Blob
later without API changes.

## Technical Context

**Language/Version**: C# 12 on .NET 8 (LTS) backend; TypeScript 5.4 (strict) on Node.js 20 LTS frontend
**Primary Dependencies**:
- Backend: ASP.NET Core 8 (controllers), EF Core 8 + `Npgsql.EntityFrameworkCore.PostgreSQL`, ASP.NET Core Identity, `Microsoft.AspNetCore.Authentication.JwtBearer`, Swashbuckle (OpenAPI), Serilog, FluentValidation
- Frontend: React 18, Vite 5, TypeScript 5.4 (strict), Tailwind CSS 3, shadcn/ui (Radix primitives), TanStack Query v5, React Router v6, react-hook-form + zod, **date-fns** v3
**Storage**: PostgreSQL 15+ (`Npgsql` provider); file attachments on local disk in Phase 1 behind an `IAttachmentStorage` abstraction (Azure Blob in a later phase)
**Testing**: Backend — xUnit + FluentAssertions (unit), `WebApplicationFactory` + `Testcontainers.PostgreSql` (integration). Frontend — Vitest + React Testing Library (unit/component), Playwright (smoke E2E for the three user stories). Manual testing checklists per user story (see `quickstart.md`).
**Target Platform**: Linux containers (backend, PostgreSQL) behind HTTPS reverse proxy; modern desktop + mobile browsers (last-two-versions Chrome/Edge/Firefox/Safari) for the SPA.
**Project Type**: Web application (frontend + backend monorepo).
**Performance Goals**: Idea-listing API ≤ 300 ms p95 at 100 concurrent users; SPA Largest Contentful Paint ≤ 1.5 s on a 4G profile; supports SC-007 (1 000 ideas, 200 users, listing page < 1.5 s p95).
**Constraints**: HTTPS only; CORS allow-list per environment; password hashing via ASP.NET Core Identity (PBKDF2); file uploads limited to 10 MB and an extension/content-type allow-list; PII never logged; bundle size budget +20 KB gzipped per added frontend dep; backend `TreatWarningsAsErrors=true`.
**Scale/Scope**: Phase 1 — 200 users, 1 000 ideas, ≤ 5 categories; ~12 REST endpoints; ~10 React routes/screens.

**Structure Decision**: Web application (Option 2 from the template) realised
as a monorepo with two top-level workspaces: `backend/` (a single .NET solution
with API/Application/Domain/Infrastructure projects + test projects) and
`frontend/` (a single Vite-based React workspace). This satisfies the
constitution's "single backend solution and a single frontend workspace"
constraint while keeping the API and SPA independently buildable and testable.
